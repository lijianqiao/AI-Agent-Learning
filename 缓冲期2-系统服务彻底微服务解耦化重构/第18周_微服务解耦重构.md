# 微服务解耦重构

## 学习

**核心思想**

- 庞然大物即将接通智能集群协作，单体烂框架寸步难行需要打散

**学习目标**

- API 规范拆解设计：将单体 Agent 系统按职责边界切割为独立服务接口
- 控制反转（IoC）与外挂插件化接口独立设计模式运用：让调度控制器不再硬编码依赖具体实现

> 缓冲期 2——前几周我们一路构建了 RAG、Memory、Tool、工作流引擎等组件，代码量已经膨胀到"牵一发而动全身"的程度。本周按下暂停键，专门做一次**架构手术**：把巨石拆成积木。

**实战与验收**

- 将所有关于搜索（底层 RAG）、查询数据库状态及 Memory 操作均强隔离独立包块
- 最终使调度控制器的圈复杂度清透洁净，完全屏蔽具体存储技术栈组件，方便单元测试及多人小组协作开发

**验收标准**

- 调度控制器只依赖抽象接口，不 import 任何具体实现；圈复杂度 ≤ 5；每个解耦模块可独立单元测试

## 第一部分学习：单体架构 vs 微服务——为什么 AI Agent 系统需要解耦

### 单体 Agent 的"初期甜蜜"与"后期地狱"

回忆前几周的建设成果——RAG 检索、Memory 管理、Tool 调用、工作流引擎、异常自愈……所有代码都挤在一个项目里，甚至可能在一个文件里互相调用。项目初期这样做很快，但一旦系统复杂度上来，就变成了**意大利面条代码**——每根面条（模块）跟其他面条缠在一起，抽一根全盘乱动。

```python
# ❌ 典型的单体 Agent 代码——一切混在一起
class MonolithAgent:
    def __init__(self):
        self.chromadb = chromadb.Client()                    # 硬绑定 ChromaDB
        self.collection = self.chromadb.get_collection("docs")
        self.redis = redis.Redis(host="localhost")           # 硬绑定 Redis
        self.pg_conn = psycopg2.connect("dbname=agent")      # 硬绑定 PostgreSQL
    
    def handle_query(self, query: str) -> str:
        # RAG 检索——直接操作 ChromaDB
        results = self.collection.query(query_texts=[query], n_results=5)
        context = "\n".join(results["documents"][0])
        
        # Memory 读取——直接操作 Redis
        history = self.redis.lrange(f"memory:{query[:20]}", 0, 10)
        memory_text = "\n".join([h.decode() for h in history])
        
        # 数据库查询——直接操作 PostgreSQL
        cur = self.pg_conn.cursor()
        cur.execute("SELECT status FROM tasks WHERE topic = %s", (query,))
        db_status = cur.fetchone()
        
        # 拼接 Prompt 并调用 LLM
        prompt = f"上下文：{context}\n历史：{memory_text}\n状态：{db_status}\n问题：{query}"
        response = call_llm(prompt)
        
        # Memory 写入——又直接操作 Redis
        self.redis.rpush(f"memory:{query[:20]}", response)
        
        return response
```

看到问题了吗？`handle_query` 这一个方法里同时操作了 ChromaDB、Redis、PostgreSQL、LLM——它既是"调度员"又是"搬运工"，圈复杂度爆炸，单元测试根本无法做（你总不能每次跑测试都启动三个数据库吧）。

### 生活类比：中央厨房 vs 分布式餐厅

**单体架构** = 一个厨师在一口大锅里炒所有菜。一开始效率高——什么都在手边，炒完一个接一个。但菜品多了之后：调料瓶互相碰撞、锅不够用、一道菜糊了整锅都得倒掉。

**微服务架构** = 把厨房拆成多个专业档口：凉菜档、热菜档、面点档、甜品档。每个档口独立运作，有自己的工具和操作台。前台（调度控制器）只需要喊"3 号桌要一个宫保鸡丁"，热菜档就动手——前台不需要知道热菜档用的是铁锅还是不粘锅。

### 单体 vs 微服务（Agent 场景对比）

| 维度 | 单体 Agent | 微服务 Agent |
| --- | --- | --- |
| 代码耦合度 | 高：改 RAG 可能影响 Memory 逻辑 | 低：每个服务边界清晰 |
| 单元测试 | 难：必须启动所有依赖（数据库等） | 易：Mock 接口即可隔离测试 |
| 团队协作 | 差：多人改同一文件冲突不断 | 好：每人负责一个服务模块 |
| 技术栈替换 | 灾难：换 ChromaDB 为 Milvus 要改遍全局 | 轻松：只改一个实现类 |
| 部署灵活性 | 全量部署，一个组件出问题全部回滚 | 独立部署，故障隔离 |
| 初期开发速度 | 快（不用设计接口） | 稍慢（需要先定义契约） |
| 后期维护成本 | 指数级增长 | 线性增长 |

### AI Agent 系统特别需要解耦的三个原因

**原因一：技术栈变化快。** 向量数据库从 ChromaDB 换到 Milvus，嵌入模型从 OpenAI 换到本地模型，Memory 从 Redis 换到 DynamoDB——如果硬耦合，每次换底层都是大手术。

**原因二：多人协作是常态。** 一个人负责 RAG 管线，一个人负责 Memory 策略，一个人负责工具集成——如果都在一个文件里改，Git 冲突就是日常。

**原因三：测试要求高。** Agent 系统的每个环节（检索、记忆、推理、执行）都需要独立验证质量，单体架构下根本无法做到精准的单元测试。

## 第二部分学习：控制反转（IoC）与依赖注入（DI）原理

### 什么是"控制反转"？

传统写法中，**高层模块主动创建低层依赖**——`MonolithAgent.__init__` 里直接 `chromadb.Client()`、`redis.Redis()` 就是典型。高层模块掌握了"用什么、怎么创建"的控制权。

**控制反转（Inversion of Control）**就是把这个控制权"反"过来——高层模块不再自己创建依赖，而是**声明"我需要一个具备 XX 能力的东西"**，由外部负责把具体实现"注入"进来。

### 生活类比：点外卖 vs 自己做饭

**没有 IoC（自己做饭）**：你想吃番茄炒蛋，于是自己去超市买番茄、买鸡蛋、买油、买盐，回家开火炒。你必须知道每种食材的品牌和来源——**你控制一切细节**。

**有了 IoC（点外卖）**：你只说"我要一份番茄炒蛋"，外卖平台（IoC 容器）负责找一家餐厅、用他们的食材和做法做好送到你手上。你不关心用的是什么牌子的番茄——**你只关心结果符合预期**。

### 依赖注入（DI）的三种姿势

依赖注入是 IoC 的一种实现方式，在 Python 中最常见的有三种：

| 注入方式 | 实现方法 | 优点 | 缺点 | 类比 |
| --- | --- | --- | --- | --- |
| 构造器注入 | `__init__(self, service: Interface)` | 依赖关系明确、不可变 | 参数列表可能变长 | 入职时签合同约定岗位职责 |
| Setter 注入 | `set_service(self, service)` | 灵活可变更 | 可能在未注入时被调用 | 在职期间调岗 |
| 接口注入 | 通过框架自动装配 | 无侵入性 | 需要框架支持 | 公司 HR 系统自动分配工位 |

在 Agent 系统中，**构造器注入**最推荐——因为 Agent 的依赖（RAG、Memory、Tools）在运行期间通常不会变，构造时一次性注入最安全。

```python
from abc import ABC, abstractmethod


# ✅ 定义抽象接口（契约）
class SearchService(ABC):
    @abstractmethod
    def search(self, query: str, top_k: int = 5) -> list[str]:
        """检索相关文档片段"""
        ...

class MemoryService(ABC):
    @abstractmethod
    def load(self, session_id: str) -> list[str]:
        """加载对话历史"""
        ...
    
    @abstractmethod
    def save(self, session_id: str, message: str) -> None:
        """保存一条对话记录"""
        ...


# ✅ 构造器注入——Agent 只依赖抽象接口
class SmartAgent:
    def __init__(
        self,
        search_service: SearchService,
        memory_service: MemoryService,
    ):
        self._search = search_service
        self._memory = memory_service
    
    def handle_query(self, session_id: str, query: str) -> str:
        context = self._search.search(query)
        history = self._memory.load(session_id)
        
        prompt = f"上下文：{context}\n历史：{history}\n问题：{query}"
        response = call_llm(prompt)
        
        self._memory.save(session_id, response)
        return response
```

现在 `SmartAgent` 完全不知道底层用的是 ChromaDB 还是 Milvus、Redis 还是 DynamoDB——它只跟**接口对话**。换底层实现？写一个新的实现类注入进去就行，`SmartAgent` 一行代码都不用改。

### IoC 前后对比

| 维度 | IoC 之前 | IoC 之后 |
| --- | --- | --- |
| Agent 知道底层实现吗 | 知道，直接 `import chromadb` | 不知道，只 `import SearchService` |
| 换数据库要改 Agent 吗 | 要改，改完还得回归测试 | 不用改，换注入即可 |
| 单元测试 | 必须启动真实数据库 | 注入 Mock 对象即可 |
| 代码可读性 | `__init__` 里一堆初始化逻辑 | `__init__` 只有参数声明 |
| 多人协作 | 容易踩到对方的初始化代码 | 各写各的实现类，互不干扰 |

## 第三部分学习：接口隔离原则（ISP）在 Agent 系统中的应用

### 什么是接口隔离原则？

**接口隔离原则（Interface Segregation Principle，ISP）**：客户端不应该被迫依赖它不使用的方法。

翻译成大白话就是：**别给我一个什么都能干的超级遥控器，给我几个各管各的小遥控器就好。**

### 生活类比：万能遥控器 vs 专用遥控器

**违反 ISP（万能遥控器）**：一个遥控器上有 200 个按钮——控制电视、空调、灯、窗帘、音响……你只想关灯，却要在 200 个按钮里找那一个。更糟的是，如果遥控器升级了"空调协议"，你的"关灯"功能也可能被波及。

**遵守 ISP（专用遥控器）**：灯光用灯光遥控器，空调用空调遥控器。每个遥控器只有它该有的按钮，简洁明了，互不干扰。

### Agent 系统中的 ISP 违规

```python
# ❌ 违反 ISP——一个"万能"接口
class AgentBackend(ABC):
    @abstractmethod
    def search_documents(self, query: str) -> list[str]: ...
    
    @abstractmethod
    def load_memory(self, session_id: str) -> list[str]: ...
    
    @abstractmethod
    def save_memory(self, session_id: str, msg: str) -> None: ...
    
    @abstractmethod
    def query_database(self, sql: str) -> list[dict]: ...
    
    @abstractmethod
    def call_external_tool(self, tool_name: str, params: dict) -> str: ...
    
    @abstractmethod
    def send_notification(self, channel: str, message: str) -> None: ...
```

问题在哪？如果你只需要一个"纯检索 Agent"，它只用 `search_documents`，但它却被迫依赖了 `send_notification`、`query_database` 等完全不相干的方法。更危险的是——改 `send_notification` 的签名，所有实现类（包括跟通知无关的）都得同步修改。

### 正确的拆法：按职责隔离接口

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass


@dataclass
class SearchResult:
    content: str
    score: float
    source: str


class SearchService(ABC):
    """职责：文档检索（RAG 底层）"""
    
    @abstractmethod
    def search(self, query: str, top_k: int = 5) -> list[SearchResult]:
        ...
    
    @abstractmethod
    def index_document(self, doc_id: str, content: str, metadata: dict) -> None:
        ...


class MemoryService(ABC):
    """职责：对话记忆管理"""
    
    @abstractmethod
    def load(self, session_id: str, limit: int = 20) -> list[str]:
        ...
    
    @abstractmethod
    def save(self, session_id: str, role: str, content: str) -> None:
        ...
    
    @abstractmethod
    def clear(self, session_id: str) -> None:
        ...


class DatabaseService(ABC):
    """职责：业务数据查询"""
    
    @abstractmethod
    def query(self, sql: str, params: tuple = ()) -> list[dict]:
        ...
    
    @abstractmethod
    def execute(self, sql: str, params: tuple = ()) -> int:
        ...


class ToolService(ABC):
    """职责：外部工具调用"""
    
    @abstractmethod
    def list_tools(self) -> list[str]:
        ...
    
    @abstractmethod
    def invoke(self, tool_name: str, params: dict) -> str:
        ...
```

现在每个接口**只管自己那一摊事**。一个纯检索 Agent 只需要依赖 `SearchService`，跟 `ToolService`、`DatabaseService` 毫无关系。

### ISP 拆分前后对比

| 维度 | 拆分前（万能接口） | 拆分后（隔离接口） |
| --- | --- | --- |
| 接口数量 | 1 个大接口，6+ 方法 | 4 个小接口，各 2-3 方法 |
| Agent 依赖 | 必须依赖全部方法 | 按需依赖，最小化 |
| 修改影响范围 | 改一个方法波及所有实现类 | 只影响相关实现类 |
| 实现复杂度 | 一个类要实现所有方法 | 每个类只实现一个接口 |
| Mock 测试 | Mock 对象需实现 6+ 方法 | Mock 对象只需 2-3 方法 |
| 团队分工 | 难以按模块分工 | 每人认领一个接口 |

## 第四部分学习：插件化架构设计

### 为什么需要插件化？

有了接口隔离，我们已经能做到"面向接口编程"。但问题来了——**具体用哪个实现，写死在代码里还是不太灵活**。比如：

- 开发环境用内存版 `InMemorySearch`，生产环境用 `ChromaDBSearch`
- 团队 A 用 Redis 做 Memory，团队 B 想用 SQLite
- 未来想支持社区贡献新的检索引擎，不改核心代码

**插件化架构**就是让"实现的注册和发现"也变成可配置的——像浏览器安装扩展一样，插上就能用，拔掉不影响主体。

### 生活类比：USB 接口

**没有插件化** = 鼠标焊死在电脑主板上。想换鼠标？拆机、拆焊、焊新的。

**插件化** = USB 接口。鼠标、键盘、U 盘都是"插件"，USB 就是"接口标准"。想换鼠标？拔掉旧的、插上新的，1 秒搞定。电脑（调度控制器）根本不关心插的是罗技还是雷蛇——只要它遵守 USB 协议（实现了接口）。

### 插件注册表设计

```python
from typing import Type


class PluginRegistry:
    """插件注册表——管理接口与实现的映射"""
    
    def __init__(self):
        self._registry: dict[type, dict[str, Type]] = {}
    
    def register(self, interface: type, name: str, impl_class: Type) -> None:
        """注册一个接口的实现插件
        
        Args:
            interface: 抽象接口类（如 SearchService）
            name: 插件名称（如 "chromadb", "milvus"）
            impl_class: 实现类（如 ChromaDBSearchService）
        """
        if interface not in self._registry:
            self._registry[interface] = {}
        self._registry[interface][name] = impl_class
    
    def get(self, interface: type, name: str) -> Type:
        """根据接口和名称获取实现类"""
        if interface not in self._registry:
            raise KeyError(f"接口 {interface.__name__} 未注册任何插件")
        if name not in self._registry[interface]:
            available = list(self._registry[interface].keys())
            raise KeyError(
                f"接口 {interface.__name__} 没有名为 '{name}' 的插件，"
                f"可用插件：{available}"
            )
        return self._registry[interface][name]
    
    def list_plugins(self, interface: type) -> list[str]:
        """列出某个接口的所有已注册插件"""
        return list(self._registry.get(interface, {}).keys())


# 全局注册表实例
plugin_registry = PluginRegistry()
```

### 用装饰器简化插件注册

```python
def register_plugin(interface: type, name: str):
    """装饰器——自动注册插件到全局注册表"""
    def decorator(cls):
        plugin_registry.register(interface, name, cls)
        return cls
    return decorator


# 使用装饰器注册插件——新增实现只需加一个装饰器
@register_plugin(SearchService, "memory")
class InMemorySearchService(SearchService):
    """内存版检索服务——用于开发/测试"""
    
    def __init__(self):
        self._docs: dict[str, str] = {}
    
    def search(self, query: str, top_k: int = 5) -> list[SearchResult]:
        results = []
        for doc_id, content in self._docs.items():
            if query.lower() in content.lower():
                results.append(SearchResult(content=content, score=1.0, source=doc_id))
        return results[:top_k]
    
    def index_document(self, doc_id: str, content: str, metadata: dict) -> None:
        self._docs[doc_id] = content


@register_plugin(SearchService, "chromadb")
class ChromaDBSearchService(SearchService):
    """ChromaDB 版检索服务——用于生产"""
    
    def __init__(self, collection_name: str = "default"):
        import chromadb
        self._client = chromadb.Client()
        self._collection = self._client.get_or_create_collection(collection_name)
    
    def search(self, query: str, top_k: int = 5) -> list[SearchResult]:
        results = self._collection.query(query_texts=[query], n_results=top_k)
        return [
            SearchResult(content=doc, score=score, source=f"chroma:{idx}")
            for idx, (doc, score) in enumerate(
                zip(results["documents"][0], results["distances"][0])
            )
        ]
    
    def index_document(self, doc_id: str, content: str, metadata: dict) -> None:
        self._collection.add(ids=[doc_id], documents=[content], metadatas=[metadata])
```

### 配置驱动的插件选择

```python
import json


def build_agent_from_config(config_path: str) -> "SmartAgent":
    """从配置文件构建 Agent——插件选择完全由配置驱动"""
    
    with open(config_path, "r") as f:
        config = json.load(f)
    
    # 示例 config.json:
    # {
    #     "search": {"plugin": "chromadb", "params": {"collection_name": "rag_docs"}},
    #     "memory": {"plugin": "redis", "params": {"host": "localhost", "port": 6379}},
    #     "database": {"plugin": "sqlite", "params": {"db_path": "agent.db"}}
    # }
    
    search_cls = plugin_registry.get(SearchService, config["search"]["plugin"])
    memory_cls = plugin_registry.get(MemoryService, config["memory"]["plugin"])
    
    search_service = search_cls(**config["search"].get("params", {}))
    memory_service = memory_cls(**config["memory"].get("params", {}))
    
    return SmartAgent(search_service=search_service, memory_service=memory_service)
```

### 插件化方案对比

| 方案 | 实现方式 | 适用场景 | 复杂度 |
| --- | --- | --- | --- |
| 硬编码 | 直接 `import` 并实例化 | 原型阶段，只有一种实现 | 最低 |
| 工厂函数 | `if-elif` 判断创建实例 | 少量固定实现（2-3 种） | 低 |
| 注册表 + 装饰器 | 本节方案，自动注册 + 配置驱动 | 多种实现、需要热插拔 | 中 |
| Entry Points（setuptools） | Python 包的标准插件机制 | 开源项目，社区贡献插件 | 较高 |
| 独立进程 + RPC | gRPC / HTTP 微服务 | 超大规模、多语言混合部署 | 最高 |

## 第五部分学习：圈复杂度降低策略

### 什么是圈复杂度？

**圈复杂度（Cyclomatic Complexity）** 由 Thomas McCabe 于 1976 年提出，衡量一段代码中**独立执行路径的数量**。简单说：if/elif/for/while/try-except 越多，圈复杂度越高。

公式：`CC = E - N + 2P`

- E = 控制流图中的边数
- N = 控制流图中的节点数
- P = 连通分量数（通常为 1）

实际开发中不用手算——工具自动检测。但经验法则很直观：

| 圈复杂度范围 | 风险等级 | 可维护性 | 类比 |
| --- | --- | --- | --- |
| 1-5 | 低 | 清晰简洁，容易理解 | 一条直路，偶尔有个路口 |
| 6-10 | 中 | 需要注意，但可控 | 城市道路，红绿灯不少 |
| 11-20 | 高 | 难以理解和测试 | 立交桥，容易走错匝道 |
| 21+ | 极高 | 几乎不可维护 | 迷宫，连导航都崩溃 |

### 生活类比：交通路口 vs 高架桥

一个方法的圈复杂度就像你开车要经过的**分岔路口数量**。如果一条路上有 20 个路口，每个路口都可能左转右转直行掉头——你迷路的概率极大。好的城市规划师会把复杂路网拆成多段简单直路，每段只有 1-2 个路口。

**降低圈复杂度 = 把复杂立交拆成多条简单直路。**

### 高圈复杂度的 Agent 代码

```python
# ❌ 圈复杂度爆炸——handle_query 中的嵌套地狱
def handle_query(self, query: str, mode: str, session_id: str = None) -> str:
    if mode == "search":
        if self.search_engine == "chromadb":
            results = self._chromadb_search(query)
        elif self.search_engine == "milvus":
            results = self._milvus_search(query)
        elif self.search_engine == "elastic":
            results = self._elastic_search(query)
        else:
            raise ValueError(f"Unknown engine: {self.search_engine}")
        
        if len(results) == 0:
            return "未找到相关内容"
        
        if session_id:
            history = self._load_memory(session_id)
            if len(history) > 10:
                history = history[-10:]
            context = "\n".join(results) + "\n" + "\n".join(history)
        else:
            context = "\n".join(results)
    
    elif mode == "database":
        try:
            db_result = self._query_db(query)
            if db_result is None:
                return "查询无结果"
            context = str(db_result)
        except Exception as e:
            if "timeout" in str(e):
                return "数据库超时，请重试"
            else:
                raise
    
    elif mode == "tool":
        # ... 更多嵌套 ...
        pass
    
    else:
        return "未知模式"
    
    return self._call_llm(context, query)
```

这段代码的圈复杂度轻松超过 15——if/elif 嵌套、异常处理、边界检查层层叠叠。

### 六大降低策略

| 策略 | 做法 | 效果 | 类比 |
| --- | --- | --- | --- |
| 策略模式 | 用接口 + 字典替代 if-elif 链 | 消除条件分支 | 把 20 个路口变成 1 个导航选择 |
| 提取方法 | 把嵌套逻辑提取为独立方法 | 每个方法只做一件事 | 把长句子拆成短句子 |
| 早返回 | 把异常/边界情况提前 return | 减少嵌套层级 | 先过安检，不合格直接劝返 |
| 依赖注入 | 让外部决定用什么实现 | 消除"选择实现"的分支 | 别人帮你点好菜，你直接吃 |
| 责任链 | 多个处理器按顺序尝试处理 | 消除 if-elif 瀑布 | 问题在各部门之间流转 |
| Guard Clause | 在方法开头做参数校验并立即返回 | 压缩主逻辑缩进 | 门卫先检查，合格再放行 |

```python
class CleanAgent:
    """重构后的 Agent——圈复杂度 ≤ 5"""
    
    def __init__(
        self,
        search_service: SearchService,
        memory_service: MemoryService,
        database_service: DatabaseService,
    ):
        self._search = search_service
        self._memory = memory_service
        self._database = database_service
        
        self._handlers: dict[str, callable] = {
            "search": self._handle_search,
            "database": self._handle_database,
        }
    
    def handle_query(self, query: str, mode: str, session_id: str = None) -> str:
        """主入口——圈复杂度降至 2（一个 if 判断）"""
        handler = self._handlers.get(mode)
        if handler is None:
            return f"未知模式：{mode}"
        
        context = handler(query, session_id)
        return self._call_llm(context, query)
    
    def _handle_search(self, query: str, session_id: str = None) -> str:
        """搜索处理——圈复杂度 3"""
        results = self._search.search(query)
        if not results:
            return "未找到相关内容"
        
        context = "\n".join(r.content for r in results)
        
        if session_id:
            history = self._memory.load(session_id, limit=10)
            context += "\n" + "\n".join(history)
        
        return context
    
    def _handle_database(self, query: str, session_id: str = None) -> str:
        """数据库处理——圈复杂度 2"""
        result = self._database.query(query)
        if not result:
            return "查询无结果"
        return str(result)
    
    def _call_llm(self, context: str, query: str) -> str:
        """调用 LLM——圈复杂度 1"""
        prompt = f"上下文：{context}\n问题：{query}"
        return f"[LLM Response based on context]"  # 实际调用 LLM
```

**关键变化**：
- `handle_query` 从 CC=15+ 降到 CC=2（只有一个 `if`）
- 具体检索引擎的选择**不在这里发生**——在构造注入时就决定了
- 每个 `_handle_*` 方法独立、可测试、圈复杂度 ≤ 3

```python
class PluginRegistry:
    def __init__(self):
        self._registry: dict[type, dict[str, Type]] = {}
    
    def register(self, interface: type, name: str, impl_class: Type) -> None:
        if interface not in self._registry:
            self._registry[interface] = {}
        self._registry[interface][name] = impl_class
    
    def get(self, interface: type, name: str) -> Type:
        if interface not in self._registry:
            raise KeyError(f"接口 {interface.__name__} 未注册任何插件")
        if name not in self._registry[interface]:
            available = list(self._registry[interface].keys())
            raise KeyError(f"插件 '{name}' 不存在，可用：{available}")
        return self._registry[interface][name]
    
    def list_plugins(self, interface: type) -> list[str]:
        return list(self._registry.get(interface, {}).keys())


registry = PluginRegistry()


def plugin(interface: type, name: str):
    """装饰器：自动注册插件"""
    def decorator(cls):
        registry.register(interface, name, cls)
        return cls
    return decorator

@plugin(SearchService, "memory")
class InMemorySearchService(SearchService):
    def __init__(self):
        self._store: dict[str, dict] = {}
    
    def search(self, query: str, top_k: int = 5) -> list[SearchResult]:
        results = []
        query_lower = query.lower()
        for doc_id, doc in self._store.items():
            content = doc["content"]
            words = query_lower.split()
            match_count = sum(1 for w in words if w in content.lower())
            if match_count > 0:
                score = match_count / len(words)
                results.append(SearchResult(content=content, score=score, source=doc_id))
        results.sort(key=lambda r: r.score, reverse=True)
        return results[:top_k]
    
    def index_document(self, doc_id: str, content: str, metadata: dict | None = None) -> None:
        self._store[doc_id] = {"content": content, "metadata": metadata or {}}


@plugin(MemoryService, "memory")
class InMemoryMemoryService(MemoryService):
    def __init__(self):
        self._sessions: dict[str, list[dict]] = {}
    
    def load(self, session_id: str, limit: int = 20) -> list[dict]:
        history = self._sessions.get(session_id, [])
        return history[-limit:]
    
    def save(self, session_id: str, role: str, content: str) -> None:
        if session_id not in self._sessions:
            self._sessions[session_id] = []
        self._sessions[session_id].append({
            "role": role,
            "content": content,
            "timestamp": time.time(),
        })
    
    def clear(self, session_id: str) -> None:
        self._sessions.pop(session_id, None)


@plugin(DatabaseService, "memory")
class InMemoryDatabaseService(DatabaseService):
    def __init__(self):
        self._tables: dict[str, list[dict]] = {}
        self._id_counter = 0
    
    def query(self, table: str, conditions: dict | None = None) -> list[dict]:
        records = self._tables.get(table, [])
        if not conditions:
            return records
        return [
            r for r in records
            if all(r.get(k) == v for k, v in conditions.items())
        ]
    
    def insert(self, table: str, record: dict) -> str:
        if table not in self._tables:
            self._tables[table] = []
        self._id_counter += 1
        record_with_id = {"id": str(self._id_counter), **record}
        self._tables[table].append(record_with_id)
        return str(self._id_counter)


@plugin(ToolService, "builtin")
class BuiltinToolService(ToolService):
    def __init__(self):
        self._tools = {
            "calculator": self._calc,
            "timestamp": self._timestamp,
            "word_count": self._word_count,
        }
    
    def list_tools(self) -> list[str]:
        return list(self._tools.keys())
    
    def invoke(self, tool_name: str, params: dict) -> ToolResult:
        start = time.time()
        if tool_name not in self._tools:
            return ToolResult(
                tool_name=tool_name,
                output=f"工具 '{tool_name}' 不存在",
                success=False,
                elapsed_ms=0,
            )
        try:
            output = self._tools[tool_name](params)
            elapsed = (time.time() - start) * 1000
            return ToolResult(tool_name=tool_name, output=output, success=True, elapsed_ms=elapsed)
        except Exception as e:
            elapsed = (time.time() - start) * 1000
            return ToolResult(tool_name=tool_name, output=str(e), success=False, elapsed_ms=elapsed)
    
    @staticmethod
    def _calc(params: dict) -> str:
        expr = params.get("expression", "0")
        allowed_chars = set("0123456789+-*/.(). ")
        if not all(c in allowed_chars for c in expr):
            return "不安全的表达式"
        return str(eval(expr))
    
    @staticmethod
    def _timestamp(params: dict) -> str:
        return str(int(time.time()))
    
    @staticmethod
    def _word_count(params: dict) -> str:
        text = params.get("text", "")
        return str(len(text))

class AgentController:
    """解耦后的 Agent 调度控制器
    
    关键特性：
    - 只依赖四个抽象接口，不 import 任何具体实现
    - 圈复杂度 ≤ 5（每个方法）
    - 完全可 Mock 测试
    """
    
    def __init__(
        self,
        search: SearchService,
        memory: MemoryService,
        database: DatabaseService,
        tools: ToolService,
    ):
        self._search = search
        self._memory = memory
        self._database = database
        self._tools = tools
    
    def handle_query(self, session_id: str, query: str) -> str:
        context_parts = []
        
        search_results = self._search.search(query, top_k=3)
        if search_results:
            context_parts.append("【检索结果】")
            for r in search_results:
                context_parts.append(f"  [{r.source}](相关度:{r.score:.2f}) {r.content}")
        
        history = self._memory.load(session_id, limit=5)
        if history:
            context_parts.append("【对话历史】")
            for msg in history:
                context_parts.append(f"  {msg['role']}: {msg['content']}")
        
        self._memory.save(session_id, "user", query)
        
        response = self._generate_response(query, "\n".join(context_parts))
        
        self._memory.save(session_id, "assistant", response)
        
        return response
    
    def query_status(self, table: str, conditions: dict | None = None) -> list[dict]:
        return self._database.query(table, conditions)
    
    def use_tool(self, tool_name: str, params: dict) -> ToolResult:
        return self._tools.invoke(tool_name, params)
    
    def _generate_response(self, query: str, context: str) -> str:
        """模拟 LLM 调用（实际项目中替换为真实 LLM）"""
        if context:
            return f"基于以下上下文回答您的问题「{query}」:\n{context}\n\n[这里是 LLM 生成的回答]"
        return f"关于「{query}」，我没有找到相关上下文，但我会尽力回答。\n[这里是 LLM 生成的回答]"
```
