# 第一周实战总结

## 一次提问的完整链路

从一次用户提问出发，完整链路是：**预算 -> 截断 -> 调用 ->（可选）计费**。

---

## 1. 预算阶段：先算可用 Token，再决定是否截断

用户输入后，先进入 `context_truncator.py` 的 `truncate_context_greedily`。

第一步不是调模型，而是做 Token 预算：

- `available_budget = max_tokens - reserve_for_output`
- 固定高优先级内容：`system_prompt + user_query`

如果固定内容已经超过预算，直接报错并终止，这是 fail-fast 的安全策略。

---

## 2. 截断阶段：贪婪保留最新历史，丢弃最旧历史

在历史对话上采用贪婪策略：

- 从**最新轮次**向前尝试保留
- 超预算时优先丢弃**最旧轮次**

这样做的目标是：在有限上下文里优先保留"最新、最相关"信息，缓解长文本退化与 Lost in the Middle。

---

## 3. 调用阶段：统一接口，不关心底层是本地还是云端

完成截断后，`ask_with_truncated_context` 会重组消息：

`[system] + kept_history + [user]`

然后调用 `llm_common.chat`。`llm_common` 负责：

- 解析模式：`local / cloud`
- 读取环境变量：`base_url`、`api_key`、默认模型
- 统一发起 `chat.completions.create`

这层抽象的价值是：上层业务只关心"输入消息"和"输出结果"，不关心底层具体是 Ollama 还是云端 API。

---

## 4. 计费阶段（可选链路）：问答与统计解耦

这里有两条链路，需要明确区分：

1. **对话执行链路**：`context_truncator -> llm_common.chat`（返回 answer/usage）  
2. **度量统计链路**：`token_count.ask_and_count`（问答 + Token 统计 + 费用估算）

也就是说，`context_truncator` 本身不会自动做完整费用统计；如果要做成本分析，需要显式走 `ask_and_count`。

---

## 5. Token 统计原则

`token_count.py` 的原则是：**usage 优先，本地估算兜底**。

- 若 API 返回 `prompt_tokens/completion_tokens`，优先使用真实值
- 若无 usage，再按模型策略本地估算

本地估算策略：

- GPT：`tiktoken`
- DeepSeek：官方 tokenizer 优先，失败后比例估算
- 本地模型（如 Ollama）：比例估算

---

## 6. 和第一周笔记的对应关系

- **Tokenization 原理** -> `token_count` 的多策略分词与估算
- **Context Window 本质** -> `max_tokens` 与 `reserve_for_output`
- **长文本退化 / 注意力分散** -> 贪婪保留最新历史、丢弃最旧历史
- **KV-Cache 成本意识** -> usage 字段与输入/输出费用拆分

---

## 7. 设计原则总结（架构视角）

从第一周的代码中提炼出五条工程原则，这些原则在后续阶段（RAG、Agent、MAS）会被反复用到：

| 原则 | 在代码中的体现 | 一句话解释 |
| --- | --- | --- |
| **Fail-fast（快速失败）** | 固定内容超预算直接 `raise ValueError`，不沉默降级 | 边界问题越早暴露越好，避免错误被掩盖后在深处爆炸 |
| **关注点分离（SRP）** | `context_truncator` / `token_count` / `llm_common` 三层各司其职 | 截断 ≠ 计费 ≠ 调用，每个模块只做一件事，改一处不牵连另外两处 |
| **策略模式（OCP）** | `TokenizerStrategy` 枚举 + `match-case` 分派不同分词器 | 对扩展开放、对修改关闭——新增模型只需在 `MODEL_SPECS` 加一行，业务代码零改动 |
| **环境隔离（12-Factor）** | `.env` 中 `LLM_MODE=local/cloud` 驱动全局模式切换 | 配置与代码分离，开发用 Ollama、生产切云端 API，代码本身不感知差异 |
| **防御性编程（Boundary Validation）** | `_validate_messages`、`_validate_history` 集中在入口校验 | 输入约束在系统边界一次性做严，业务逻辑内部不重复防御，保持主流程干净 |

这五条原则与软件工程经典理论的映射：

```
Fail-fast            →  让错误尽快可见（Martin Fowler《重构》）
关注点分离            →  SOLID 的 S：单一职责原则
策略模式 / OCP        →  SOLID 的 O：开闭原则
环境隔离             →  12-Factor App 第三条：Config（配置与代码分离）
防御性编程            →  Clean Architecture：边界层承担输入校验职责
```

> **学习价值**：这套原则不只适用于 LLM 调用层。到第 18 周做微服务解耦重构时，你会发现同样的五条原则在更大规模的系统里依然成立——只是粒度从"函数/模块"变成了"服务/组件"。

---

## 一句话总括

**先做预算治理，再做上下文裁剪，再做统一模型调用，最后按需进入 Token 与成本度量闭环。**  
这就是从"能跑"到"可控、可解释、可优化"的 Agent 工程化路径。
