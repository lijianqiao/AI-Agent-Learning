## 第一部分学习：商业 API vs 开源模型成本 Trade-off

### 为什么要做模型选型？

一个生产级 AI 系统，每天可能要处理成千上万条请求。如果所有请求都发给最贵的模型（如 GPT-4o），成本会失控；如果全用最便宜的模型，复杂任务的质量又会下降。

**架构师的核心任务：让对的请求用对的模型，花最少的钱，干最多的事。**

### 两类模型的核心差异

| 维度 | 商业 API（GPT-4o/Claude 3.5） | 开源本地（Qwen/Llama3） |
| --- | --- | --- |
| 调用方式 | HTTP 请求，按 Token 计费 | 本地部署，按算力计费 |
| 推理质量 | 顶级，尤其复杂推理 | 中等至良好（7B-70B 差异大） |
| 延迟 | 受网络影响，通常 1-5s | 取决于硬件，GPU 好则很快 |
| 数据隐私 | 数据发往第三方服务器 | 数据留在本地，完全可控 |
| 扩展成本 | 线性增长（QPS 越高越贵） | 固定硬件成本，QPS 低时贵 |
| 适用场景 | 复杂推理、创意写作、高准确率要求 | 高频简单任务、隐私敏感场景 |

### 成本估算示例

以每天 10,000 次请求为例：

**场景A：全用 GPT-4o**

```
平均每次输入：500 tokens，输出：200 tokens
输入费用：10,000 × 500 / 1,000,000 × $5 = $25/天
输出费用：10,000 × 200 / 1,000,000 × $15 = $30/天
日成本：$55/天 ≈ $1,650/月
```

**场景B：路由分流（70% 简单任务用 Qwen，30% 复杂任务用 GPT-4o）**

```
GPT-4o 部分（3,000次）：$16.5/天
Qwen API 部分（7,000次）：约 $2/天（约 1/10 价格）
日成本：$18.5/天 ≈ $555/月
```

**节省：约 66%，远超验收标准的 30%目标。**

### 路由决策的四个维度

1. **任务复杂度**：简单问答 vs 多步推理
2. **延迟要求**：实时交互（<1s）vs 异步处理（可接受3s+）
3. **隐私级别**：敏感数据必须走本地模型
4. **准确率要求**：金融/医疗高准确率场景 vs 普通聊天场景

## 第二部分学习：LLM 网关架构与 LiteLLM

### 为什么需要 LLM 网关？

没有网关时，代码直接调用各家 API：

```python
# 乱糟糟的直接调用
if provider == "openai":
    from openai import OpenAI
    client = OpenAI(api_key=OPENAI_KEY)
    # ... OpenAI 格式
elif provider == "anthropic":
    import anthropic
    client = anthropic.Anthropic(api_key=ANTHROPIC_KEY)
    # ... Anthropic 格式（完全不同！）
elif provider == "qwen":
    # 又是另一套...
```

**每家 API 格式不同，换个模型要重写代码，维护噩梦。**

### LLM 网关的核心价值

```
你的应用代码
     ↓（统一的 OpenAI 格式）
  LLM 网关（LiteLLM）
     ↓ 自动转换格式
  ├── OpenAI GPT-4o
  ├── Anthropic Claude
  ├── 阿里 Qwen
  ├── 本地 Ollama
  └── 任意其他模型
```

**一套代码，所有模型。**

### LiteLLM 核心能力

#### 1. 统一接口调用

```python
import litellm

# 调用 GPT-4o
response = litellm.completion(
    model="gpt-4o",
    messages=[{"role": "user", "content": "你好"}]
)

# 调用 Claude —— 代码完全一样，只改 model 参数！
response = litellm.completion(
    model="claude-3-5-sonnet-20241022",
    messages=[{"role": "user", "content": "你好"}]
)

# 调用本地 Ollama
response = litellm.completion(
    model="ollama/qwen2.5:7b",
    messages=[{"role": "user", "content": "你好"}]
)
```

#### 2. API Key 轮询容灾

```python
# 配置多个 Key，自动在失败时切换
litellm.api_key_list = [
    "sk-key1-openai",
    "sk-key2-openai",
    "sk-key3-openai",
]
# 某个 Key 限流或失效时，自动切换到下一个
```

#### 3. 备用模型（Fallback）

```python
response = litellm.completion(
    model="gpt-4o",
    messages=[...],
    fallbacks=["claude-3-5-sonnet-20241022", "ollama/qwen2.5:72b"],
    # GPT-4o 挂了就自动试 Claude，再挂就试本地模型
)
```

## 第三部分学习：语义路由（Semantic Routing）

### 什么是语义路由？

**语义路由 = 根据用户请求的"意图和复杂度"，自动决定用哪个模型处理。**

不是简单的关键词匹配，而是用 Embedding 向量相似度来判断请求属于哪类任务，再映射到对应的模型配置。

### 生活类比：医院分诊台

你去医院，不是每个人都直接去看主任医师。分诊台护士会先评估：

- **头疼发烧**：普通科室，挂普通号（便宜快速）
- **疑难杂症**：转专家门诊（贵但准确）
- **心脏骤停**：直接急诊（最高优先级）

语义路由就是 AI 系统里的分诊台。

### 路由规则设计示例

## 第四部分学习：熔断限流机制

### 为什么需要熔断限流？

没有保护机制时，一旦某个 API 变慢或挂掉：

- 大量请求堆积等待超时
- 超时请求触发重试，流量翻倍
- 重试又失败，形成**雪崩效应**，整个系统崩溃

**熔断器就是系统的"保险丝"，过载时主动断开，保护整体稳定。**

### 熔断器的三种状态

```
                  失败次数超阈值
     [关闭状态] ─────────────────→ [开启状态]
   （正常放行）                   （直接拒绝）
         ↑                              ↓
         │           超时后             │
         │    ←──────────────── [半开状态]
         │   （放行少量试探请求）
         │   试探成功则恢复
```

### 限流算法对比

| 算法 | 原理 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 固定窗口 | 每分钟最多N次，超出拒绝 | 简单 | 窗口边界有突刺 |
| 滑动窗口 | 滚动统计最近N秒的请求数 | 平滑 | 内存占用稍高 |
| 令牌桶 | 以固定速率发令牌，有令牌才能请求 | 允许突发流量 | 实现稍复杂 |
| 漏桶 | 请求排队，以固定速率处理 | 绝对平滑 | 不允许突发，延迟高 |

**推荐使用令牌桶**：允许短时突发（比如批量上传），同时长期保持平均速率不超限。

### 三大方案一览

#### vLLM

**定位**：高吞吐量推理服务，适合生产级高并发场景

**核心技术**：PagedAttention——把 KV-Cache 分页管理（类似操作系统的虚拟内存），显存利用率提升 3-4 倍，吞吐量相比原生 HuggingFace 提升 **10-24 倍**。

**适用场景**：有自己 GPU 服务器、QPS 高（>10）、需要最大化推理吞吐的团队

```bash

```

#### TGI（Text Generation Inference）

**定位**：HuggingFace 官方推理服务，生产级稳定性好

**核心特性**：支持量化（4bit/8bit）、流式输出、多 GPU 张量并行

**适用场景**：已在 HuggingFace 生态中的团队，需要稳定官方支持

#### Ollama

**定位**：开发者本地运行模型的最简方案

**核心特性**：一条命令拉起模型，自动管理量化和内存，有完整的模型库

**适用场景**：本地开发调试、小团队低成本部署、对运维要求低的场景


## 实战：LiteLLM 网关 + 语义路由完整实现

```python
import litellm
import numpy as np
from openai import OpenAI
from typing import Optional

# ===== 语义路由器 =====
class SemanticRouter:
    """
    基于 Embedding 余弦相似度的语义路由器。
    将用户请求路由到最匹配的模型配置。
    """

    ROUTE_CONFIG = {
        "simple_qa": {
            "examples": [
                "帮我翻译这段话", "今天几号", "Python list 和 tuple 的区别",
                "这个单词什么意思", "帮我格式化这段 JSON",
            ],
            "model": "ollama/qwen2.5:7b",
            "max_tokens": 512,
            "label": "简单问答",
        },
        "complex_reasoning": {
            "examples": [
                "帮我设计一个分布式系统架构", "分析这份财报的投资价值",
                "调试这段有竞态条件的多线程代码", "写一份完整的技术方案",
            ],
            "model": "gpt-4o",
            "max_tokens": 4096,
            "label": "复杂推理",
        },
    }

    def __init__(self):
        self.embed_client = OpenAI()
        self._build_index()

    def _embed(self, text: str) -> np.ndarray:
        resp = self.embed_client.embeddings.create(
            model="text-embedding-3-small",
            input=text,
        )
        return np.array(resp.data[0].embedding)

    def _build_index(self):
        """预计算所有示例的 Embedding，启动时一次性完成"""
        self.index = {}
        for route_name, config in self.ROUTE_CONFIG.items():
            embeddings = [self._embed(ex) for ex in config["examples"]]
            self.index[route_name] = np.stack(embeddings)
        print("语义路由索引构建完成。")

    def _cosine_similarity(self, a: np.ndarray, b: np.ndarray) -> float:
        return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b) + 1e-9))

    def route(self, query: str, threshold: float = 0.4) -> dict:
        """
        对 query 进行路由，返回最匹配的路由配置。
        """
        query_emb = self._embed(query)
        best_route = "complex_reasoning"  # 默认走高能模型，保底准确率
        best_score = -1.0

        for route_name, example_embs in self.index.items():
            # 计算 query 与该路由所有示例的最大相似度
            scores = [self._cosine_similarity(query_emb, ex_emb) for ex_emb in example_embs]
            max_score = max(scores)
            if max_score > best_score:
                best_score = max_score
                best_route = route_name

        config = self.ROUTE_CONFIG[best_route].copy()
        config["route_name"] = best_route
        config["confidence"] = round(best_score, 4)

        # 低置信度时保守地走高能模型
        if best_score < threshold:
            config = self.ROUTE_CONFIG["complex_reasoning"].copy()
            config["route_name"] = "complex_reasoning_fallback"
            config["confidence"] = round(best_score, 4)

        return config


# ===== LiteLLM 网关封装 =====
class LLMGateway:
    """
    统一 LLM 调用网关：语义路由 + 熔断重试 + 费用统计
    """

    def __init__(self):
        self.router = SemanticRouter()
        self.call_stats = {"total": 0, "by_model": {}}

    def _update_stats(self, model: str, input_tokens: int, output_tokens: int):
        self.call_stats["total"] += 1
        if model not in self.call_stats["by_model"]:
            self.call_stats["by_model"][model] = {"calls": 0, "input_tokens": 0, "output_tokens": 0}
        stats = self.call_stats["by_model"][model]
        stats["calls"] += 1
        stats["input_tokens"] += input_tokens
        stats["output_tokens"] += output_tokens

    def chat(self, user_query: str, verbose: bool = True) -> str:
        # 1. 语义路由
        route = self.router.route(user_query)

        if verbose:
            print(f"\n[路由决策] → {route['label']} | 模型: {route['model']} | 置信度: {route['confidence']}")

        # 2. 通过 LiteLLM 统一调用（带自动重试）
        response = litellm.completion(
            model=route["model"],
            messages=[{"role": "user", "content": user_query}],
            max_tokens=route["max_tokens"],
            num_retries=3,        # 失败自动重试3次
            timeout=30,           # 超时30秒
            fallbacks=["gpt-4o-mini"] if "ollama" in route["model"] else [],
        )

        # 3. 统计费用数据
        usage = response.usage
        self._update_stats(route["model"], usage.prompt_tokens, usage.completion_tokens)

        return response.choices[0].message.content

    def cost_report(self) -> str:
        """输出费用对比报告"""
        # 简化费用估算（实际应用中使用 litellm.completion_cost）
        MODEL_PRICES = {
            "gpt-4o": {"input": 5.0, "output": 15.0},
            "gpt-4o-mini": {"input": 0.15, "output": 0.6},
            "ollama/qwen2.5:7b": {"input": 0.0, "output": 0.0},  # 本地无API费用
        }

        lines = ["\n===== 费用统计报告 ====="]
        total_cost = 0.0
        gpt4o_hypothetical = 0.0

        for model, stats in self.call_stats["by_model"].items():
            price = MODEL_PRICES.get(model, {"input": 1.0, "output": 3.0})
            cost = (stats["input_tokens"] / 1e6 * price["input"] +
                    stats["output_tokens"] / 1e6 * price["output"])
            total_cost += cost
            # 假设全用 GPT-4o 的成本
            gpt4o_hypothetical += (stats["input_tokens"] / 1e6 * 5.0 +
                                   stats["output_tokens"] / 1e6 * 15.0)
            lines.append(
                f"  {model}: {stats['calls']}次 | "
                f"输入{stats['input_tokens']}tokens | 实际费用 ${cost:.4f}"
            )

        savings_pct = (1 - total_cost / max(gpt4o_hypothetical, 0.0001)) * 100
        lines.append(f"\n  实际总费用：${total_cost:.4f}")
        lines.append(f"  若全用GPT-4o：${gpt4o_hypothetical:.4f}")
        lines.append(f"  节省比例：{savings_pct:.1f}%")
        return "\n".join(lines)


# ===== 测试用例 =====
if __name__ == "__main__":
    gateway = LLMGateway()

    test_queries = [
        "帮我把'你好世界'翻译成英文",               # → 简单，走廉价模型
        "分析微服务架构和单体架构的技术选型依据",    # → 复杂，走GPT-4o
        "Python 的 list comprehension 怎么用？",    # → 简单，走廉价模型
        "帮我设计一套 AI Agent 的 Memory 分层架构", # → 复杂，走GPT-4o
    ]

    for q in test_queries:
        answer = gateway.chat(q)
        print(f"Q: {q[:40]}...")
        print(f"A: {answer[:100]}...\n")

    print(gateway.cost_report())
```

## 第五部分学习：推理模型（Reasoning Models）调度策略

### 什么是推理模型？

2025 年起，以 OpenAI o3/o4-mini、DeepSeek-R1 为代表的**推理模型**成为 AI Agent 架构中的重要组件。它们与普通生成模型（GPT-4o、Claude 4 Sonnet）有本质区别：

| 维度 | 普通生成模型（GPT-4o） | 推理模型（o3 / DeepSeek-R1） |
| --- | --- | --- |
| **思考方式** | 直接输出答案 | 内部生成"思维链"（Chain of Thought），再输出最终答案 |
| **Token 消耗** | 仅消耗输入+输出 tokens | 额外消耗"思考 tokens"（不可见但计费） |
| **延迟** | 通常 1-3s | 5-30s（思考越深越慢） |
| **擅长场景** | 创意写作、摘要、问答 | 数学证明、代码调试、多步规划、逻辑推理 |
| **成本** | 中等 | 高（但复杂任务成功率高，失败重试成本反而更低） |

### 生活类比：快思考 vs 慢思考

**普通模型 = 脱口秀演员**：听到问题立刻说出答案，反应极快，但遇到复杂逻辑题就容易出错（"脑子里没有草稿纸"）。

**推理模型 = 数学考生**：先在草稿纸上演算步骤，花更多时间，但最终答案准确率大幅提升（内部 CoT 就是那张草稿纸）。

### 推理模型的三个关键参数

```python
# OpenAI o3-mini 的推理预算控制
response = client.chat.completions.create(
    model="o3-mini",
    messages=[{"role": "user", "content": "证明勾股定理"}],
    # 推理预算：low / medium / high
    # low: 快速、省钱，适合简单逻辑
    # high: 深度思考，适合复杂推理，成本高 3-5 倍
    reasoning_effort="medium",
)

# DeepSeek-R1 的思考 token 控制
response = client.chat.completions.create(
    model="deepseek-reasoner",
    messages=[{"role": "user", "content": "分析这段代码的时间复杂度"}],
    # 可通过 max_tokens 间接限制思考深度
    max_tokens=4096,
)
# 返回结果中包含 reasoning_content（思维链）和 content（最终答案）
print(response.choices[0].message.reasoning_content)  # 思考过程
print(response.choices[0].message.content)             # 最终答案
```

### 架构师的调度策略：三层路由

在语义路由基础上，增加第三层——**推理复杂度路由**：

```
用户请求
    ↓
【第一层：意图路由】
    ├── 闲聊/简单问答 → 本地 Qwen3 4B（$0）
    ├── 知识检索/摘要 → GPT-4o-mini / Claude Haiku（低成本）
    └── 需要深度推理 → 【第二层：推理复杂度评估】
                            ├── 中等复杂（代码审查、简单逻辑）→ o4-mini [reasoning_effort=low]
                            └── 高度复杂（数学证明、多步规划）→ o3 [reasoning_effort=high]
                                                                     / DeepSeek-R1
```

### 何时该用推理模型？决策清单

| 场景 | 推荐模型 | 理由 |
| --- | --- | --- |
| 数学计算 / 证明 | o3 / DeepSeek-R1 | 需要多步推导 |
| 复杂代码调试 | o4-mini / DeepSeek-R1 | 需要追踪状态变化 |
| 多跳逻辑推理 | o3-mini (medium) | 需要维护中间结论 |
| 创意写作 / 摘要 | GPT-4o / Claude 4 Sonnet | 推理模型在此无优势但成本高 |
| 简单问答 / 分类 | Qwen3 4B / Llama3 | 本地跑，零成本 |
| **Agent 规划任务** | o4-mini / DeepSeek-R1 | **Agent 最核心使用场景** |

### Token Budget 控制的重要性

推理模型有一个陷阱：**"过度思考"**——给一个简单问题，模型可能生成 5000 个思考 token，而实际答案只需要 50 个输出 token。

```python
# ❌ 错误用法：不控制 Token Budget
response = client.chat.completions.create(
    model="o3",
    messages=[{"role": "user", "content": "1+1等于几？"}],
    # 可能消耗数千思考 tokens，成本极不划算
)

# ✅ 正确用法：简单问题用 low reasoning_effort
response = client.chat.completions.create(
    model="o3-mini",
    messages=[{"role": "user", "content": "1+1等于几？"}],
    reasoning_effort="low",  # 限制思考深度，省钱
)

# ✅ 复杂问题才开 high
response = client.chat.completions.create(
    model="o3",
    messages=[{"role": "user", "content": "设计一个分布式事务的解决方案..."}],
    reasoning_effort="high",
)
```

### 整合进路由网关

```python
# 在 SemanticRouter 基础上增加推理复杂度判断
class ReasoningAwareRouter(SemanticRouter):
    def route(self, query: str) -> tuple[str, dict]:
        """返回 (model_name, extra_params)"""
        base_model = super().route(query)

        # 如果路由到高配模型，进一步判断是否需要推理模型
        if base_model == "gpt-4o" and self._needs_reasoning(query):
            complexity = self._estimate_complexity(query)
            if complexity == "high":
                return "o3", {"reasoning_effort": "high"}
            else:
                return "o4-mini", {"reasoning_effort": "low"}

        return base_model, {}

    def _needs_reasoning(self, query: str) -> bool:
        """启发式判断：包含数学/证明/步骤/为什么 等关键词"""
        reasoning_keywords = ["证明", "推导", "为什么", "分析", "步骤", "计算", "优化算法"]
        return any(kw in query for kw in reasoning_keywords)

    def _estimate_complexity(self, query: str) -> str:
        """简单启发式：长问题/多约束 → high"""
        return "high" if len(query) > 200 else "medium"
```

### 推理模型 vs 普通模型：面试必考点

**Q：推理模型的"思考 tokens"是什么？**
A：推理模型在生成最终答案前，会先在内部生成一段推理过程（类似人类打草稿）。这段推理不会出现在输出中，但会消耗 tokens 并计费。OpenAI 称之为 "reasoning tokens"，DeepSeek-R1 通过 `<think>` 标签暴露出来。

**Q：什么场景下推理模型反而不划算？**
A：简单问答、摘要生成、风格改写——这些任务普通模型已能高质量完成，推理模型的额外思考 tokens 纯属浪费。

**Q：如何在 Agent 架构中合理使用推理模型？**
A：仅在 Agent 的"规划层"（Planner）使用推理模型制定任务分解方案，在"执行层"（Executor）的工具调用、文本生成等步骤使用普通模型，平衡质量与成本。
