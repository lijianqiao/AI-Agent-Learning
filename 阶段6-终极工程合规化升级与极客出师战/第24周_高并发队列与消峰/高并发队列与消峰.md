# 高并发队列与消峰

## 学习

**学习目标**

- 当在线推理 QPS 激增，如何保护核心网不崩塌？掌握消息队列异步消峰的核心架构，让系统在流量洪峰下依旧稳定。

> 本周承接前序推理服务部署成果，正式进入"高并发防护"阶段——用消息中间件把同步推理请求变成异步排队处理，让系统在 QPS 暴增时优雅降级而非直接崩溃。

**实战**

- 将前台轮询拉取模型改作依靠消息中间件通道（Kafka / RabbitMQ）排队执行机制，释放前端锁区，后台常驻线程池异步消峰处理回吐

**验收标准**

- 在 QPS 突增 5 倍的压测场景下，系统不崩溃，队列积压自动消化，无请求永久丢失

## 第一部分学习：同步 vs 异步架构在 AI 推理场景的差异

### 为什么 AI 推理天然不适合同步架构？

传统 Web 请求（查数据库、读缓存）响应时间通常在 10-100ms 量级。但 AI 推理（尤其是大模型）单次请求动辄 1-30 秒，这意味着一个同步线程会被**长时间占住**，吞吐量急剧下降。

### 生活类比：餐厅点餐模式

**同步架构 = 柜台现做汉堡**：每个顾客到柜台点餐，厨师当场做，做完才叫下一个。一旦排队人数超过厨师处理速度，队伍堵到门外，后面的人全部等着，甚至直接走人（超时断连）。

**异步架构 = 取号排队制**：你到柜台点餐拿到一个号码牌（任务 ID），然后去旁边坐着等。厨房按自己的节奏做，做完广播叫号。前台永远不堵，顾客体验也好（可以刷手机等通知）。

### 同步 vs 异步架构对比

| 维度 | 同步架构 | 异步架构 |
| --- | --- | --- |
| 请求处理方式 | 客户端发请求，等推理完成才返回 | 客户端发请求拿到任务 ID，后续轮询或回调获取结果 |
| 线程占用 | 推理期间线程被锁死，无法服务其他请求 | 请求入队即释放线程，工作线程池独立消费 |
| 峰值承压 | QPS 超过线程池上限即拒绝服务 | 队列充当缓冲区，可承受远超处理能力的瞬时流量 |
| 超时风险 | 大模型推理 > 30s 易触发 HTTP 超时 | 客户端不等待，无超时问题 |
| 实现复杂度 | 简单，一个 API 搞定 | 需引入消息队列、任务状态管理、结果回传通道 |
| 适用场景 | 推理 < 500ms 的轻量模型 | 推理 > 1s 的大模型、批量推理、高并发场景 |

### 同步架构崩溃的典型链路

```
QPS 突增 5x
    ↓
所有工作线程被推理任务占满（每个推理耗时 5-10s）
    ↓
新请求无线程可用，排队等待
    ↓
等待队列溢出 → 连接超时 → 客户端重试
    ↓
重试请求进一步加重负载（重试风暴）
    ↓
系统彻底崩溃（雪崩）
```

这就是经典的**雪崩效应**（Cascading Failure）。异步消峰架构的核心目标就是**在链路中插入一个弹性缓冲区**，打断这条崩溃链路。

## 第二部分学习：消息队列核心概念

### 消息队列是什么？

消息队列（Message Queue）是一种**先进先出（FIFO）的中间件**，负责在生产者和消费者之间传递消息，解耦两端的速度差异。

### 生活类比：快递驿站

你（生产者）寄快递，不需要等收件人（消费者）在家。快递员把包裹放到驿站（消息队列），收件人什么时候方便什么时候来取。

- **快递驿站** = 消息队列（缓冲区）
- **寄件人** = Producer（生产者）
- **收件人** = Consumer（消费者）
- **包裹** = Message（消息）
- **驿站的货架** = Topic / Queue（主题/队列）
- **多个收件人分不同货架取** = Consumer Group（消费者组）

### 四大核心概念

| 概念 | 含义 | AI 推理场景中的角色 |
| --- | --- | --- |
| **Producer** | 消息生产者，发送消息到队列 | API 网关收到推理请求后，将请求体序列化为消息发送到队列 |
| **Consumer** | 消息消费者，从队列拉取并处理消息 | 后台推理 Worker，从队列取任务、调用模型、写回结果 |
| **Topic / Queue** | 消息的逻辑分组通道 | 按模型类型或优先级分设不同 Topic，如 `inference-gpt4`、`inference-fast` |
| **Broker** | 消息代理服务器，负责存储和转发 | Kafka 集群或 RabbitMQ 服务端 |

### 消息传递模式

| 模式 | 描述 | 类比 |
| --- | --- | --- |
| **点对点（P2P）** | 一条消息只被一个消费者消费 | 快递包裹只有一个收件人 |
| **发布/订阅（Pub/Sub）** | 一条消息可被多个消费者消费 | 广播通知，所有人都能听到 |
| **消费者组（Consumer Group）** | 组内竞争消费，组间广播 | 同一个部门只派一个人去取件，但每个部门都要取一份 |

在 AI 推理消峰场景中，我们主要用**点对点 + 消费者组**模式：多个推理 Worker 竞争消费同一个队列，实现负载均衡。

## 第三部分学习：Kafka vs RabbitMQ 选型对比

### 两大主流消息队列

**Kafka** 和 **RabbitMQ** 是生产环境中最常见的两个消息中间件，但它们的设计哲学完全不同。

### 生活类比

**Kafka = 高速公路收费站**：车流量巨大，按车道（Partition）分流，所有车都留下通行记录（持久化日志），事后可以回放任意时间段的通行数据。

**RabbitMQ = 邮局**：寄信人把信投进邮筒，邮局负责分拣、投递到收件人信箱。信被取走就没了（ACK 后删除），流程精确可控。

### 核心对比

| 维度 | Kafka | RabbitMQ |
| --- | --- | --- |
| 设计哲学 | 分布式日志流平台 | 传统消息代理（AMQP 协议） |
| 消息模型 | 基于 Partition 的有序日志，消费者通过 Offset 读取 | 基于 Queue 的推/拉模式，ACK 后消息删除 |
| 吞吐量 | 极高（百万级 QPS） | 中等（万级 QPS） |
| 延迟 | 毫秒级（但有批量攒批机制） | 微秒到毫秒级（实时性更好） |
| 消息回溯 | 支持，可回放任意 Offset 的历史消息 | 不支持，ACK 后消息删除 |
| 消息路由 | 简单，按 Topic + Partition | 强大，支持 Direct/Fanout/Topic/Header 四种交换器 |
| 消息优先级 | 不支持 | 支持 |
| 死信队列 | 需自行实现 | 原生支持 |
| 运维复杂度 | 高（依赖 ZooKeeper/KRaft） | 中等 |
| 适用场景 | 大数据流处理、日志收集、高吞吐场景 | 任务队列、RPC、需要精确路由的业务场景 |

### AI 推理场景怎么选？

| 场景特征 | 推荐 | 原因 |
| --- | --- | --- |
| 单模型推理，QPS < 1万 | RabbitMQ | 运维简单，延迟低，原生支持死信队列和优先级 |
| 多模型推理，QPS > 1万 | Kafka | 吞吐量高，Partition 天然支持多 Worker 并行 |
| 需要重放失败请求 | Kafka | 消息持久化日志，可回溯任意 Offset |
| 需要任务优先级（VIP 用户优先推理） | RabbitMQ | 原生消息优先级支持 |
| 已有 Kafka 基础设施 | Kafka | 技术栈统一，减少运维成本 |

> 对于大多数中小规模 AI 推理服务，**RabbitMQ 是更实用的起步选择**；当吞吐量需求超过万级 QPS 或需要消息回溯能力时，再迁移到 Kafka。

## 第四部分学习：异步消峰架构设计

### 整体架构

```
客户端（前端/调用方）
    │
    │  POST /inference  {"prompt": "..."}
    ↓
┌──────────────────┐
│   API 网关层      │  ← 接收请求，生成 task_id，入队，立即返回 202 Accepted
│  （FastAPI）      │
└────────┬─────────┘
         │ 消息入队
         ↓
┌──────────────────┐
│   消息队列        │  ← 缓冲区，承接流量洪峰
│ （Kafka/RabbitMQ）│
└────────┬─────────┘
         │ 消费者拉取
         ↓
┌──────────────────┐
│  推理 Worker 池   │  ← 常驻进程，按自身能力匀速消费
│ （asyncio 协程池）│
│                  │
│  Worker 1        │
│  Worker 2        │
│  Worker 3  ...   │
└────────┬─────────┘
         │ 写回结果
         ↓
┌──────────────────┐
│   结果存储        │  ← Redis / DB，存储推理结果
│  （Redis）        │
└──────────────────┘
         ↑
         │  GET /result/{task_id}  轮询或 WebSocket 推送
         │
客户端（前端/调用方）
```

### 关键设计决策

| 决策点 | 方案 | 理由 |
| --- | --- | --- |
| 请求入队后返回什么？ | 返回 `202 Accepted` + `task_id` | 释放前端连接，客户端凭 task_id 查询结果 |
| 结果怎么通知客户端？ | 轮询 / WebSocket / 回调 | 轮询最简单；WebSocket 实时性好；回调适合服务间调用 |
| Worker 怎么控制并发？ | 固定大小的协程池 / 信号量 | 防止 Worker 同时拉取过多任务导致 GPU OOM |
| 队列积压怎么办？ | 监控队列深度，动态扩缩 Worker | 积压 > 阈值时告警，自动扩容 Worker 实例 |
| 请求丢失怎么办？ | ACK 机制 + 死信队列 | 处理失败的消息进入死信队列，不会永久丢失 |

### 请求全生命周期

```
1. 客户端发送推理请求 → API 收到
2. API 生成唯一 task_id（UUID）
3. API 将 {task_id, prompt, timestamp} 序列化为消息，发送到队列
4. API 立即返回 {task_id, status: "queued"} 给客户端（HTTP 202）
5. 推理 Worker 从队列拉取消息
6. Worker 调用模型推理
7. Worker 将结果写入 Redis：task_id → {status: "done", result: "..."}
8. Worker 向队列发送 ACK（确认消费成功）
9. 客户端轮询 GET /result/{task_id}，拿到结果
```

## 第五部分学习：背压（Backpressure）与流量控制

### 什么是背压？

**背压（Backpressure）** 是指当下游处理能力不足时，向上游发出"慢一点"的信号，防止系统被压垮。

### 生活类比：水管系统

想象一根水管，入水口（Producer）的水流量突然变大，但出水口（Consumer）口径固定：

- **没有背压** = 水管爆裂（系统崩溃）
- **有背压** = 水管中间加了一个蓄水池（消息队列），水池满了就关小入水口阀门（限流）

### 背压策略对比

| 策略 | 实现方式 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **无限缓冲** | 队列无上限，消息堆积 | 不丢消息 | 内存/磁盘可能撑爆 |
| **有界队列 + 拒绝** | 队列满时拒绝新消息（返回 503） | 保护系统不崩溃 | 客户端需要处理拒绝逻辑 |
| **有界队列 + 丢弃旧消息** | 队列满时丢弃最早的消息 | 保证新鲜度 | 可能丢失重要请求 |
| **速率限制（Rate Limiting）** | 在 API 层限制入队速率 | 最前端保护 | 需要合理设置阈值 |
| **动态扩缩容** | 监控队列深度，自动增加 Worker | 弹性应对 | 扩容有延迟，不适合毫秒级突增 |

### AI 推理场景的推荐背压方案

```
                    ┌─────────────────────┐
                    │  第一层：API 限流     │  Token Bucket / 滑动窗口
                    │  QPS > 阈值时返回 429│
                    └──────────┬──────────┘
                               ↓
                    ┌─────────────────────┐
                    │  第二层：有界队列     │  maxlen = N（如 10000）
                    │  队列满时返回 503    │
                    └──────────┬──────────┘
                               ↓
                    ┌─────────────────────┐
                    │  第三层：Worker 信号量│  semaphore = M（如 GPU 数 × 2）
                    │  控制同时推理任务数   │
                    └──────────┬──────────┘
                               ↓
                    ┌─────────────────────┐
                    │  第四层：自动扩缩容   │  队列深度 > 阈值 → 拉起新 Worker
                    │  队列深度监控告警     │
                    └─────────────────────┘
```

**分层防御**的理念：每一层都有自己的防护职责，即使某一层被击穿，下一层依然能兜底。

## 第六部分学习：死信队列与请求不丢失保障

### 消息丢失的三个环节

```
Producer → [Broker] → Consumer

 环节①      环节②      环节③
 发送丢失   存储丢失    消费丢失
```

| 丢失环节 | 原因 | 解决方案 |
| --- | --- | --- |
| **①发送丢失** | 网络抖动导致消息未到达 Broker | Producer 确认机制（Kafka: `acks=all`；RabbitMQ: `publisher confirm`） |
| **②存储丢失** | Broker 宕机，内存中消息未落盘 | 消息持久化（Kafka: 副本机制；RabbitMQ: `durable=True`） |
| **③消费丢失** | Worker 取到消息但处理失败/崩溃 | 手动 ACK + 死信队列（处理失败不确认，消息重新入队） |

### 死信队列（Dead Letter Queue, DLQ）

**死信队列**是专门存放"无法正常消费的消息"的队列。当一条消息满足以下条件之一时，会被路由到死信队列：

1. **消费失败次数超过阈值**（如重试 3 次仍失败）
2. **消息过期**（TTL 到期未被消费）
3. **队列满了被拒绝**

### 生活类比：退信机制

你寄一封信（消息），邮局尝试投递 3 次（重试），收件人始终不在家（消费失败）。邮局不会把信扔掉，而是盖上"无法投递"的戳，退回到一个专门的退信箱（死信队列）。你可以定期检查退信箱，决定是重新寄还是人工处理。

### 死信队列工作流

```
正常队列（inference_queue）
    │
    │ Worker 消费
    ↓
处理成功？
    ├── 是 → ACK 确认，消息删除
    └── 否 → 重试（最多 3 次）
                │
                └── 3 次都失败 → 消息路由到死信队列（inference_dlq）
                                    │
                                    ↓
                              运维人员/自动脚本检查
                                    │
                                    ├── 可恢复 → 修复后重新入队
                                    └── 不可恢复 → 记录日志 + 告警
```

### 请求不丢失的完整保障链

| 层次 | 机制 | 作用 |
| --- | --- | --- |
| **入队前** | 数据库预写（WAL） | 请求入队前先写数据库，即使入队失败也能从 DB 恢复 |
| **队列内** | 消息持久化 + 副本 | Broker 宕机不丢消息 |
| **消费时** | 手动 ACK | 只有处理成功才确认，失败自动重试 |
| **重试耗尽** | 死信队列 | 捕获所有无法处理的消息 |
| **最终兜底** | 定时对账 | 定期比对入队数和消费数，发现差异即补偿 |

## 第七部分学习：实战代码——基于 asyncio 的异步推理队列系统

> 以下代码使用纯 Python asyncio 实现一个**最小可运行的异步推理队列系统**，不依赖外部 Kafka/RabbitMQ，用 `asyncio.Queue` 模拟消息队列行为，便于本地学习和验证。生产环境中替换为真实消息中间件即可。

### 7.1 完整系统代码

```python
import asyncio
import uuid
import time
import random
from dataclasses import dataclass, field
from typing import Optional
from enum import Enum


# ============================================================
# 数据结构定义
# ============================================================

class TaskStatus(Enum):
    QUEUED = "queued"
    PROCESSING = "processing"
    DONE = "done"
    FAILED = "failed"
    DEAD = "dead"  # 进入死信队列


@dataclass
class InferenceTask:
    task_id: str
    prompt: str
    created_at: float
    retry_count: int = 0
    max_retries: int = 3
    status: TaskStatus = TaskStatus.QUEUED
    result: Optional[str] = None
    error: Optional[str] = None


# ============================================================
# 模拟 AI 推理函数（实际场景替换为模型调用）
# ============================================================

async def mock_inference(prompt: str) -> str:
    """模拟AI推理，随机耗时1-3秒，10%概率失败"""
    await asyncio.sleep(random.uniform(1.0, 3.0))
    if random.random() < 0.1:
        raise RuntimeError(f"模型推理异常: GPU OOM on prompt '{prompt[:20]}...'")
    return f"[推理结果] 基于 '{prompt[:30]}' 的回答: 这是模型生成的内容_{random.randint(1000,9999)}"


# ============================================================
# 异步推理队列系统
# ============================================================

class AsyncInferenceQueue:
    def __init__(
        self,
        max_queue_size: int = 10000,
        max_workers: int = 5,
        max_retries: int = 3,
    ):
        self.main_queue: asyncio.Queue = asyncio.Queue(maxsize=max_queue_size)
        self.dead_letter_queue: asyncio.Queue = asyncio.Queue()
        self.results: dict[str, InferenceTask] = {}
        self.max_workers = max_workers
        self.max_retries = max_retries

        # 统计指标
        self.stats = {
            "submitted": 0,
            "completed": 0,
            "failed": 0,
            "dead_lettered": 0,
            "rejected": 0,
        }

    # ---------- Producer 端：提交推理请求 ----------

    async def submit(self, prompt: str) -> Optional[str]:
        """提交推理任务，返回 task_id；队列满时返回 None（模拟 503）"""
        task_id = str(uuid.uuid4())[:8]
        task = InferenceTask(
            task_id=task_id,
            prompt=prompt,
            created_at=time.time(),
            max_retries=self.max_retries,
        )

        try:
            self.main_queue.put_nowait(task)
            self.results[task_id] = task
            self.stats["submitted"] += 1
            return task_id
        except asyncio.QueueFull:
            self.stats["rejected"] += 1
            return None

    # ---------- Consumer 端：Worker 消费循环 ----------

    async def worker(self, worker_id: int):
        """单个推理 Worker 的消费循环"""
        while True:
            task: InferenceTask = await self.main_queue.get()
            task.status = TaskStatus.PROCESSING
            print(f"  [Worker-{worker_id}] 开始处理 {task.task_id}: '{task.prompt[:25]}...'")

            try:
                result = await mock_inference(task.prompt)
                task.status = TaskStatus.DONE
                task.result = result
                self.stats["completed"] += 1
                print(f"  [Worker-{worker_id}] ✓ 完成 {task.task_id}")
            except Exception as e:
                task.retry_count += 1
                task.error = str(e)

                if task.retry_count < task.max_retries:
                    task.status = TaskStatus.QUEUED
                    await self.main_queue.put(task)
                    print(f"  [Worker-{worker_id}] ✗ 失败 {task.task_id}，第{task.retry_count}次重试入队")
                else:
                    task.status = TaskStatus.DEAD
                    await self.dead_letter_queue.put(task)
                    self.stats["dead_lettered"] += 1
                    self.stats["failed"] += 1
                    print(f"  [Worker-{worker_id}] ☠ 死信 {task.task_id}，重试耗尽，进入死信队列")

            self.main_queue.task_done()

    # ---------- 查询结果 ----------

    def get_result(self, task_id: str) -> Optional[dict]:
        """查询任务状态和结果"""
        task = self.results.get(task_id)
        if not task:
            return None
        return {
            "task_id": task.task_id,
            "status": task.status.value,
            "result": task.result,
            "error": task.error,
            "retry_count": task.retry_count,
        }

    # ---------- 死信队列处理 ----------

    async def process_dead_letters(self):
        """定期处理死信队列中的任务"""
        recovered = 0
        while not self.dead_letter_queue.empty():
            task: InferenceTask = await self.dead_letter_queue.get()
            print(f"  [DLQ] 检查死信 {task.task_id}: {task.error}")
            task.retry_count = 0
            task.status = TaskStatus.QUEUED
            await self.main_queue.put(task)
            recovered += 1
        if recovered:
            print(f"  [DLQ] 已恢复 {recovered} 条死信消息重新入队")
        return recovered

    # ---------- 启动系统 ----------

    async def start_workers(self) -> list[asyncio.Task]:
        """启动 Worker 池"""
        workers = []
        for i in range(self.max_workers):
            w = asyncio.create_task(self.worker(i))
            workers.append(w)
        print(f"[系统] 已启动 {self.max_workers} 个推理 Worker")
        return workers

    def print_stats(self):
        """打印系统运行指标"""
        print("\n" + "=" * 50)
        print("         系统运行指标")
        print("=" * 50)
        print(f"  提交总数:   {self.stats['submitted']}")
        print(f"  完成总数:   {self.stats['completed']}")
        print(f"  失败总数:   {self.stats['failed']}")
        print(f"  死信总数:   {self.stats['dead_lettered']}")
        print(f"  拒绝总数:   {self.stats['rejected']}")
        print(f"  队列积压:   {self.main_queue.qsize()}")
        print(f"  死信积压:   {self.dead_letter_queue.qsize()}")
        print("=" * 50)
```

### 7.2 代码关键设计解读

| 设计点 | 实现方式 | 为什么这样做 |
| --- | --- | --- |
| 队列有界 | `asyncio.Queue(maxsize=10000)` | 防止无限积压撑爆内存（背压第二层） |
| 队列满时拒绝 | `put_nowait()` 捕获 `QueueFull` | 模拟生产环境返回 503 |
| Worker 数量固定 | `max_workers=5` | 模拟 GPU 资源有限，不能无限并发推理 |
| 失败重试 | `retry_count < max_retries` 时重新入队 | 瞬时错误（如 GPU OOM）可能重试成功 |
| 死信队列 | 重试耗尽后进入 `dead_letter_queue` | 不丢弃任何请求，保留人工处理的机会 |
| 结果存储 | `dict` 模拟 Redis | 客户端通过 `task_id` 查询结果 |
| `task_done()` | 每次消费完调用 | 配合 `queue.join()` 实现等待所有任务完成 |

### 7.4 扩展：接入真实 RabbitMQ 的 Producer/Consumer

> 以下代码展示如何将上面的系统接入真实 RabbitMQ，使用 `aio-pika` 库。

```python
import aio_pika
import json
import asyncio


RABBITMQ_URL = "amqp://guest:guest@localhost:5672/"
QUEUE_NAME = "inference_queue"
DLQ_NAME = "inference_dlq"


# ============================================================
# Producer：接入 RabbitMQ 发送推理任务
# ============================================================

async def rabbitmq_producer(prompt: str, task_id: str):
    """将推理请求发送到 RabbitMQ"""
    connection = await aio_pika.connect_robust(RABBITMQ_URL)
    async with connection:
        channel = await connection.channel()

        # 声明死信交换器和死信队列
        dlx = await channel.declare_exchange("inference_dlx", aio_pika.ExchangeType.DIRECT)
        dlq = await channel.declare_queue(DLQ_NAME, durable=True)
        await dlq.bind(dlx, routing_key="dead")

        # 声明主队列，配置死信路由
        queue = await channel.declare_queue(
            QUEUE_NAME,
            durable=True,
            arguments={
                "x-dead-letter-exchange": "inference_dlx",
                "x-dead-letter-routing-key": "dead",
                "x-message-ttl": 300000,  # 5 分钟 TTL
            },
        )

        message_body = json.dumps({
            "task_id": task_id,
            "prompt": prompt,
            "timestamp": asyncio.get_event_loop().time(),
        })

        await channel.default_exchange.publish(
            aio_pika.Message(
                body=message_body.encode(),
                delivery_mode=aio_pika.DeliveryMode.PERSISTENT,
            ),
            routing_key=QUEUE_NAME,
        )
        print(f"[Producer] 已发送任务 {task_id} 到 RabbitMQ")


# ============================================================
# Consumer：从 RabbitMQ 消费推理任务
# ============================================================

async def rabbitmq_consumer(worker_id: int):
    """从 RabbitMQ 消费推理任务"""
    connection = await aio_pika.connect_robust(RABBITMQ_URL)
    async with connection:
        channel = await connection.channel()
        await channel.set_qos(prefetch_count=1)

        queue = await channel.declare_queue(QUEUE_NAME, durable=True)

        async with queue.iterator() as queue_iter:
            async for message in queue_iter:
                async with message.process(requeue=True):
                    body = json.loads(message.body.decode())
                    task_id = body["task_id"]
                    prompt = body["prompt"]

                    print(f"[Worker-{worker_id}] 收到任务 {task_id}")
                    try:
                        result = await mock_inference(prompt)
                        print(f"[Worker-{worker_id}] ✓ 完成 {task_id}: {result[:50]}...")
                        # 实际场景：将结果写入 Redis
                    except Exception as e:
                        print(f"[Worker-{worker_id}] ✗ 失败 {task_id}: {e}")
                        raise  # requeue=True 会自动重新入队
```

## 第八部分学习：队列监控与自动扩缩容

### 核心监控指标

| 指标 | 含义 | 告警阈值（参考） |
| --- | --- | --- |
| **队列深度（Queue Depth）** | 当前队列中等待处理的消息数 | > 5000 告警，> 8000 紧急 |
| **消费速率（Consume Rate）** | 每秒消费的消息数 | 持续低于入队速率时告警 |
| **消费延迟（Consumer Lag）** | 最新消息 Offset - 消费者当前 Offset | > 10000 告警 |
| **处理耗时（P95 Latency）** | 单条消息从入队到处理完成的时间 | > 60s 告警 |
| **死信增长率** | 死信队列新增消息速率 | > 0 就需要关注 |
| **Worker 利用率** | 活跃 Worker 数 / 总 Worker 数 | > 90% 考虑扩容 |

### 自动扩缩容策略

```python
import asyncio


class AutoScaler:
    """基于队列深度的自动扩缩容器"""

    def __init__(
        self,
        inference_system,
        check_interval: int = 10,
        scale_up_threshold: int = 100,
        scale_down_threshold: int = 10,
        min_workers: int = 2,
        max_workers: int = 20,
    ):
        self.system = inference_system
        self.check_interval = check_interval
        self.scale_up_threshold = scale_up_threshold
        self.scale_down_threshold = scale_down_threshold
        self.min_workers = min_workers
        self.max_workers = max_workers
        self.current_workers = inference_system.max_workers
        self.worker_tasks: list[asyncio.Task] = []

    async def monitor_loop(self):
        """持续监控队列深度并调整 Worker 数量"""
        while True:
            queue_depth = self.system.main_queue.qsize()
            dlq_depth = self.system.dead_letter_queue.qsize()

            print(f"\n[AutoScaler] 队列深度={queue_depth}, 死信={dlq_depth}, Workers={self.current_workers}")

            if queue_depth > self.scale_up_threshold and self.current_workers < self.max_workers:
                new_count = min(self.current_workers + 2, self.max_workers)
                print(f"[AutoScaler] ⬆ 扩容: {self.current_workers} → {new_count}")
                for i in range(self.current_workers, new_count):
                    task = asyncio.create_task(self.system.worker(i))
                    self.worker_tasks.append(task)
                self.current_workers = new_count

            elif queue_depth < self.scale_down_threshold and self.current_workers > self.min_workers:
                new_count = max(self.current_workers - 1, self.min_workers)
                print(f"[AutoScaler] ⬇ 缩容: {self.current_workers} → {new_count}")
                if self.worker_tasks:
                    removed = self.worker_tasks.pop()
                    removed.cancel()
                self.current_workers = new_count

            if dlq_depth > 0:
                print(f"[AutoScaler] ⚠ 发现 {dlq_depth} 条死信，触发自动恢复")
                await self.system.process_dead_letters()

            await asyncio.sleep(self.check_interval)
```
